# 直播文字稿整理

## 核心原则

直播文字稿整理采用“双层结构”：

- 场次整理卡负责保留细节，避免因为摘要而漏掉操作、作业、面试建议和风险提醒。
- 主题 wiki 负责长期查询，把多场直播中反复出现的技术、项目和求职内容沉淀成可复用知识。

不要只做求职知识库，也不要只做逐场摘要。正确顺序是：先保真，再抽取；先场次，后主题。

## 资料层级

```text
raw/live/
  原始直播文字稿，默认不改写。

wiki/直播课/场次/
  每场一篇场次整理卡，按单个文件整理。

_meta/live/直播课-场次索引.md
  记录所有直播文件、整理状态、主题标签和沉淀去向。

_meta/live/直播课术语校正表.md
  记录 ASR 误识别、专有名词和待核实术语。

wiki/
  长期技术主题页、求职主题页、项目/作业主题页。
```

## 单文件整理步骤

1. 读取 `wiki/00-索引.md` 和 `_meta/live/直播课-场次索引.md`，了解已有结构。
2. 确认待整理的原始文字稿位置；正式摄取时优先复制或保存到 `raw/live/`。
3. 读取一个直播文件，不同时处理多个文件。
4. 判断文字稿来源、倍速情况、视频是否可回看、时间轴可信度。
5. 创建或更新对应场次整理卡。
6. 按时间线切分主题段，提取技术概念、操作步骤、求职信息、作业、风险和待核实点。
7. 更新 `_meta/live/直播课-场次索引.md`。
8. 必要时更新 `_meta/live/直播课术语校正表.md`。
9. 如果本场已经出现可复用主题，把内容链接到已有主题页；如果主题尚未成熟，先记录在场次卡中。
10. 按“日期 + 直播整理任务”更新 `_meta/00-维护日志.md`；同一天同一批直播整理合并到同一条，不把每个小修正写成独立日志。

## 主题沉淀节奏

- 每整理 1 个文件：更新场次卡和场次索引。
- 每整理 3-5 个文件：回看是否需要合并或更新主题 wiki。
- 不急着为每个术语建页面；先用场次卡承接细节。
- 主题页必须保留来源，最好链接到具体场次和时间段。
- 只有沉淀出长期主题页时，才更新 `wiki/00-索引.md`；普通场次整理只更新 `_meta/live/直播课-场次索引.md`。

## 防漏检查

整理每个时间段时，至少检查这些问题：

- 有没有技术概念、工具、框架、库、平台名？
- 有没有操作步骤、命令、配置、账号、环境变量、部署信息？
- 有没有面试题、简历表达、项目包装、求职路径？
- 有没有作业、验收标准、下次课安排？
- 有没有风险、安全、合规、资金、账号、私钥相关提醒？
- 有没有讲者判断、事实信息、我的整理结论和待核实内容？
- 有没有画面依赖，例如“这里点一下”“这个配置”“这个页面”但文字稿缺失屏幕信息？

## 时间轴规则

时间轴只作为定位线索，不默认等于原视频时间。

- 原视频字幕导出：时间轴可信度通常为高。
- 原速录音转写：时间轴可信度通常为中。
- 倍速录屏或倍速录音转写：时间轴可信度通常为低或中，需要记录播放倍速。

如果是倍速录制转写，换算关系是：

```text
原视频时间 = 文字稿时间 × 播放倍速
```

## 风险标注规则

涉及账号、支付、资金、交易所、私钥、跨境、法律或合规判断的内容，不默认隐藏，但必须标注。

推荐标记：

```md
> 风险标记: 涉及账号、支付、资金、交易所、私钥、跨境或合规内容。以下内容仅作为课程记录，执行前需自行核实风险。
```

具体条目可以使用：

```md
- [风险/资金] ...
- [风险/账号] ...
- [待核实/合规] ...
```
