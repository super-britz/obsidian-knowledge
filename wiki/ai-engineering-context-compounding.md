# AI 工程化：上下文工程与知识复利

## 核心观点

这篇来源把 speckit、openspec 等规范驱动 AI 编程工具的落地挫折，归纳为一个更大的问题：AI 编程的关键瓶颈不只是“规范不够清楚”，而是企业研发中的上下文、历史经验、平台约束和执行过程知识没有被系统化管理。

作者提出的替代思路是 AI 工程化：用 `context/` 管理高信号上下文，用 Agent/Skill/Command 承接工作流，用 MCP 连接外部系统，并通过复合工程让每次实践都降低下一次类似工作的成本。

## 背景

来源中的团队以存量后端项目迭代为主，真实需求往往跨服务、跨团队、跨平台和跨流程。一个“商品发放”需求可能同时涉及产品确认、运营玩法、外部接口、配置中心、平台商品配置、数据库变更和合规约束。

在这种场景里，speckit 的 `Constitution -> Specify -> Plan -> Tasks -> Implement` 流程，以及 openspec 的提案、Delta、归档机制，都有清晰的工程化价值，但容易假设需求可以被一次性描述、线性推进和稳定命名。企业现实则更接近动态协作、并行推进、持续调整和反复“考古”。

## 规范驱动工具的局限

### speckit 的主要问题

- 流程过于理想化：假设需求清晰、可一次性规划、可按阶段线性推进。
- 缺少考古能力：不能自动关联历史坑点、现有能力、配置规范、平台知识和协作约束。
- 知识不会沉淀：实施过程中的排查信息、经验教训和决策依据没有自然进入下一次工作。
- 宪章系统偏僵化：适合标准化项目，但在历史债务、内部框架和合规约束很多的企业场景里成本较高。

### openspec 的主要问题

- Delta 机制假设变更可以被稳定提案化，但真实需求常常多线并行、动态调整。
- Fail-Fast 可以保证一致性，也可能把复杂冲突交回给人手工判断。
- 语义命名是脆弱点：产品、运营、研发可能用不同词描述同一件事。
- Archive 更像合并，不等于学习；它更新 spec，但不一定把执行上下文和经验教训沉淀下来。

## AI 工程化的解法

AI 工程化不是否定规范驱动，而是把规范放进更大的系统里：规范只是上下文的一部分，真正要管理的是工作场景、执行过程和经验复利。

它的核心组件可以理解为：

- `context/`：沉淀业务背景、服务概要、技术约束、平台规范和历史经验。
- `requirements/`：管理进行中的需求状态，让需求可以阶段性推进和调整。
- `workspace/`：承载临时分析、代码检索、服务加载和执行过程材料。
- `Command`：用户表达意图的入口。
- `Agent`：做意图识别、流程路由、任务拆分和上下文选择。
- `Skill`：执行可复用的固定流程，例如需求创建、方案生成、环境搭建和提交代码。
- `MCP`：连接 TAPD、代码托管、团队 wiki、配置平台等外部系统。

## 上下文工程

来源强调，上下文工程不是把更多内容塞进模型窗口，而是维护一个最小但高信号的上下文集合。

有效上下文应当遵循几条原则：

- 分层组织：用服务概要、业务概要、模块概要、接口概要帮助 AI 快速定位。
- Just-in-Time：只在需要时加载相关上下文，避免上下文窗口被长期污染。
- 概要加代码指针：概要帮助定位，细节直接读代码，不维护一份和代码一样详细的 spec。
- 压缩执行经验：把排查路径、配置坑、平台规则和协作约束结构化记录下来。
- 控制工具噪音：工具和 Skill 越多，越需要路由机制避免上下文污染。

这里的关键判断是：详细 spec 可能变成“另一份代码”。如果 spec 详细到足以精确指导实现，就会带来 spec 和 code 的双重维护成本；如果 spec 太简略，又难以真正约束 AI。AI 工程化倾向于用分层概要和代码指针避免这个悖论。

## 复合工程

复合工程关注边际成本是否递减。传统规范驱动工具帮助“把当前事情做完”，但下一次类似需求仍可能从人脑重新开始。AI 工程化希望每次工作都把知识留在工具链里。

来源中的循环可以概括为：

```text
Plan -> Work -> Review -> Compound
```

其中 `Compound` 是关键：把执行过程中新发现的坑点、配置规则、平台经验、协作约束和检查清单沉淀到 `context/` 或 `Skill` 中。第一次类似需求成本可能较高，后续需求可以复用上下文，最终把高频流程封装成能力单元。

## 多 Agent 与状态传递

复杂任务如果依赖单个 Agent 长时间持有全部上下文，容易遇到窗口爆满、上下文腐蚀和行为漂移。来源提出的方向是主 Agent + Subagent：

- 主 Agent 负责意图识别、任务拆分和结果汇总。
- Subagent 使用独立上下文窗口处理原子任务。
- 通过结构化状态文件传递结果，而不是依赖单一对话记忆。

这种架构更像团队协作：不同角色各自专注，通过文档交接，而不是要求一个人连续记住所有事情。

## 与 LLM Wiki 模式的关系

这篇来源和 [[llm-wiki-pattern]] 可以互相补充。LLM Wiki 模式解决的是个人知识库如何把 raw 资料持续整合进 wiki；AI 工程化解决的是研发工作流如何把历史经验持续整合进工具链。

两者共享同一个原则：原始材料不等于可复用知识，只有经过整理、链接、索引和复盘，知识才会在下一次工作中变成上下文。
